## Problem

doxygen-guard produces sequence diagrams that look structurally valid but are semantically wrong on close inspection. The root cause is that the tagging model conflates three distinct relationships between functions and requirements:

1. **Fulfills** — the function directly implements the requirement's behavior (the chef cooking the steak)
2. **Supports** — the function is shared infrastructure that multiple requirements depend on (the knife in the kitchen)
3. **Assumes** — the function's sequence starts after another requirement is already satisfied (the device is paired before OTA can run)

Currently only `@req` (fulfills) and `@utility` (exempt/invisible) exist. This forces two bad choices:

- Tag infrastructure with `@req` → it appears in feature diagrams, and inbound caller detection pulls the entire call tree of every caller into the diagram (the "waiter's full shift in the kitchen" problem)
- Tag infrastructure with `@utility` → it's invisible everywhere, hiding real dependencies

### Evidence from self-dogfood

REQ-CONFIG-001 ("YAML config loading and merging") diagram shows:
- `validate_output_path` tagged `@req REQ-CONFIG-001` — a utility used by config, trace, AND impact
- Inbound caller `run_precommit` pulled in because it calls `validate_output_path`
- `run_precommit`'s full edge set bleeds in: git_add, collect_changed_functions, run_trace, format_json — none related to config loading

11 of 16 self-dogfood diagrams exhibit this pattern.

### Evidence from d_linux_bissell_iot_midea

307 generated puml files, massive duplication. Hub functions like DCI dispatch handle 40+ event types. The human-authored diagrams (90+ puml files) avoid this by only drawing feature-specific sequences — infrastructure is assumed.

### The hub function problem

Central dispatch functions (run_precommit, DCI event loops) touch many requirements. Any tagging scheme produces large doxygen blocks on these functions. This is an accepted cost — the doxygen block IS the documentation, the hook enforces freshness on every commit, and the alternative (separate puml files) rots.

## Proposed Tags

### `@supports REQ-XXX`

Declares that a function is shared infrastructure backing one or more requirements, without being part of their sequence flow.

```c
/**
 * @brief Reject output paths with traversal or absolute components.
 * @version 1.2
 * @utility
 * @supports REQ-CONFIG-001
 * @supports REQ-TRACE-001
 * @supports REQ-IMPACT-003
 */
void validate_output_path(const char *path) {
```

**Behavior:**
- Function does NOT appear in REQ-CONFIG-001's feature sequence diagram
- Function DOES appear in a separate infrastructure dependency view (per-REQ or system-wide)
- `@utility` + `@supports` = "I'm infrastructure that backs these REQs"
- `@supports` without `@utility` = function has its own `@req` for one feature but also supports others

### `@assumes REQ-XXX`

Declares that a function's sequence starts after another requirement is satisfied. Rendered as a cross-reference in the diagram header, not as drawn edges.

```c
/**
 * @brief Handle cloud OTA notification.
 * @version 1.0
 * @req REQ-OTA-001
 * @assumes REQ-PAIR-001
 * @handles EVENT:CLOUD_OTA_AVAILABLE
 */
void OTA_OnUpdateAvailable(const char *version, const char *url) {
```

**Behavior:**
- Diagram header includes: `Preconditions: REQ-PAIR-001 (Device Pairing)`
- No edges drawn for the assumption — it's a pointer to another diagram
- Reader knows "see REQ-PAIR-001 for how we got to the state where this sequence starts"

## Diagram Generation Changes

### Feature diagrams (existing, improved)

- Only functions with `@req REQ-XXX` (fulfills) appear as sequence participants
- `@supports` functions excluded from feature diagrams entirely
- `@assumes` rendered as header cross-references
- Inbound caller detection scoped: only emit the edge FROM caller TO target function, do NOT run full edge builder on inbound callers

### Infrastructure overview diagram (new)

- Generated per-REQ or system-wide (config option)
- Shows which `@supports` functions back which requirements
- Shows which `@assumes` relationships exist between requirements
- Non-sequential — dependency graph, not sequence diagram
- Could be a PlantUML component diagram or a simple dependency matrix

## Impact on Existing Tags

| Tag | Current behavior | New behavior |
|-----|-----------------|--------------|
| `@req` | Function appears in REQ diagram with full edge scan | Same, but inbound callers scoped to direct edges only |
| `@utility` | Exempt from req coverage, invisible to tracer | Same, but can optionally declare `@supports` |
| `@internal` | Same as utility | Same as utility |
| `@supports` | N/A (new) | Declares infrastructure backing without sequence participation |
| `@assumes` | N/A (new) | Declares precondition cross-reference in diagram header |
| `@emits` | Produces async edge to handler | No change |
| `@handles` | Receives async edge from emitter | No change |
| `@ext` | Produces sync call edge | No change |
| `@triggers` | Produces note on diagram | No change |

## Edge Scoping Fixes (independent of new tags)

Two fixes needed in `build_sequence_edges`, both required regardless of whether new tags are adopted:

### Fix A: Inbound caller scoping

The current `_find_inbound_callers` + `build_sequence_edges` treats inbound callers as first-class emitters, running the full edge builder on them. Inbound callers must produce ONLY edges to the target REQ's functions — no emit edges, ext edges, or call edges to unrelated functions.

### Fix B: Call edge REQ filtering

`_build_call_edges` scans the body for ALL tagged functions and produces edges to every one it finds. For a hub function like `run_precommit` (direct `@req REQ-VAL-001` participant), this means edges to `run_trace()`, `collect_changed_functions()`, `format_json()` — none of which are REQ-VAL-001 functions. The call edge builder must filter targets to only functions that share a `@req` tag with the current diagram's REQ, or that are direct `@handles`/`@ext` targets of the emitter.

Without Fix B, Fix A alone is insufficient — direct REQ participants still bleed their full call tree into the diagram.

### Fix C: @supports exclusion from edge targets

Functions tagged `@supports REQ-XXX` (but not `@req REQ-XXX`) must be excluded as call edge targets for REQ-XXX's diagram. `@supports` declares backing without sequence participation — the function should not appear as an edge source or target in feature diagrams.

## Migration Path

1. **Phase 1: Edge scoping fixes (A+B)** — immediate, no new tags needed, fixes diagram pollution for both inbound callers and direct participants
2. **Phase 2: Implement `@supports` and `@assumes`** — new tag parsing, diagram rendering changes, Fix C
3. **Phase 3: Audit self-dogfood `@req` tags** — reclassify infrastructure functions as `@utility` + `@supports`
4. **Phase 4: Generate infrastructure overview diagrams** — markdown dependency table (not PlantUML)

Phases 1-3 can ship as patch releases. Phase 4 is a minor version feature.

## Validation Criteria

The proposal succeeds if:

1. REQ-CONFIG-001 self-dogfood diagram shows ONLY: load_config → deep_merge → validate_config_schema flow. No git_add, no run_trace, no format_json.
2. d_linux_bissell_iot_midea diagrams match the scope and clarity of the human-authored puml files — feature-specific sequences without infrastructure bleed.
3. Hub functions (run_precommit, DCI dispatch) have large but accurate doxygen blocks that the hook enforces on every commit.
4. Infrastructure dependencies are visible in a separate view, not lost.
5. Cross-requirement assumptions (OTA assumes pairing) are explicit and navigable.

## Open Questions (Resolved by Adversarial Analysis)

- **`@supports` required or optional on `@utility`?** — Optional. Pure helpers (string formatting, logging wrappers) stay `@utility` without `@supports`. Only infrastructure that meaningfully backs a requirement needs `@supports`. Forcing it on every utility creates noise without value.
- **Infrastructure overview format?** — Markdown dependency table, not PlantUML. A component diagram adds rendering complexity for minimal benefit. A table of "utility → REQs it supports" is scannable and diffable.
- **Hub function `@handles` grouping syntax?** — No wildcard syntax. Decompose hub functions into sub-dispatchers instead (the d_linux_bissell_iot_midea pattern). Hub functions that dispatch 40+ events should delegate to smaller functions that each handle a subset. This is a code quality improvement that also solves the tagging problem.
