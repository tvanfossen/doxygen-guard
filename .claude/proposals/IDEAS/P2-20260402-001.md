# doxygen-guard LSP Server

## Problem

LLM sessions and human developers must run the pre-commit hook to discover violations. This is a feedback loop measured in seconds-to-minutes. An LSP server provides real-time feedback, completions, and diagnostics inline — before the hook ever runs.

The README is the only "guide" for AI consumers. An LSP server makes the tool's knowledge available programmatically — any client (Claude Code, Copilot, Cursor, VS Code, Neovim) gets the same guidance without reading docs.

## What It Would Provide

### Diagnostics (real-time violations)
- Missing `@brief`, `@version`, `@return` — as you type
- Version staleness — when body changes without version bump
- Unknown tags with Levenshtein suggestions
- Missing `@req` on non-exempt functions
- File-level block missing `@file`/`@brief`/`@version`

### Completions
- `@req ` → completion list from requirements file (REQ-0010, REQ-0020, ...)
- `@emits ` → completion list from known event constants
- `@handles ` → same
- `@ext ` → completion list from `module::function` pairs across the project
- `@module ` → completion from existing module names

### Hover
- Hover `@req REQ-0010` → shows requirement name, description, acceptance criteria
- Hover `@ext comm_manager::Comm_SendStatus` → shows function brief + return type
- Hover `@emits EVENT_SENSOR_DATA_READY` → shows which functions handle this event

### Code Actions
- "Add missing @return tag" — inserts with AST-derived type
- "Add missing @version tag" — inserts `@version 1.0`
- "Bump @version" — increments existing version
- "Add file-level doxygen block" — inserts template with @file/@brief/@version/@module

### Workspace Symbols
- All tagged functions indexed by REQ, module, participant
- Navigate to any function tagged with a given requirement

## Architecture

```
┌─────────────────┐     ┌──────────────────┐
│  Editor / LLM   │────▶│  doxygen-guard    │
│  (LSP client)   │◀────│  LSP server       │
└─────────────────┘     └────────┬─────────┘
                                 │ shares
                        ┌────────▼─────────┐
                        │  doxygen_guard    │
                        │  (existing lib)   │
                        │  parser, checks,  │
                        │  config, tracer   │
                        └──────────────────┘
```

- Long-running process (started by editor or `doxygen-guard --lsp`)
- Shares parser, checks, config loader with the hook
- Watches `.doxygen-guard.yaml` and requirements files for changes
- Indexes all source files on workspace open, updates incrementally on save

## Dependencies

- `pygls` (Python LSP framework) or `lsprotocol` (lower-level)
- Everything else is already in `doxygen_guard.*`

## Effort Estimate

- **Phase 1**: Diagnostics only (reuse check_presence, check_tags, check_version_staleness) — 1-2 sessions
- **Phase 2**: Completions from requirements file and tag corpus — 1 session
- **Phase 3**: Hover with requirement metadata — 1 session
- **Phase 4**: Code actions for quick fixes — 1 session

## Distribution

- Same PyPI package: `pip install doxygen-guard[lsp]` (extras_require for pygls)
- Entry point: `doxygen-guard --lsp` or `doxygen-guard-lsp`
- VS Code extension wrapper (thin — just points at the LSP binary)
- Claude Code: available via MCP or hooks integration
